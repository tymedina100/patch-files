Everyone wants to build “AI agents” right now.

But a lot of teams are still tripping over the basics:

- Is this actually an agent, or just a workflow?

### LinkedIn Post 2 — The vocabulary is the architecture
The fastest way to make an AI agent project more grounded:

Get the team aligned on the words.

“Memory” is not the same as “context.”  
“Workflow” is not the same as “agent.”  
“Guardrails” are not a substitute for good permissions.  
“Multi-agent” does not automatically mean better.  
“Bigger context window” does not mean better context.

These distinctions sound small until you are deciding whether a system can send emails, change customer records, run code, or spend money.

So I made a one-page AI agents glossary for builders.

It covers the core terms in plain language: tool calling, structured outputs, RAG, embeddings, orchestration, ReAct loops, HITL, sandboxing, prompt injection, tracing, evals, MCP, latency/cost budgets, and more.

The rule of thumb: ship the smallest reliable agentic system — narrow permissions, explicit state, validated tool calls, strong traces, bounded loops, and human approval for risky actions.
